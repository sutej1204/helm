markdown# Build: 4-Way Invoice Match (RECON #1)

## Context
Helm is a financial operations platform for mid-market wholesale distributors. The app already has a `/matching-engine` route handling three-way matching (PO + Receipt + Invoice). This task adds the fourth leg — matching invoices against the underlying Special Pricing Agreement (SPA) — to catch contract-price violations before payment. Flagged variances route to a Recovery queue.

## Goal
Ship a demoable 4-way match feature in one sprint. Works end-to-end on seeded data. No live ERP integration in scope.

## Follow existing design system
Use the current Helm app shell, tokens, components (tables, pills, cards, breadcrumbs, app chrome, tabs). Match the visual language of `/matching-engine`. Do not introduce new patterns.

## Scope — four screens

### 1. AP Inbox · `/ap-inbox`
Table of incoming invoices. Columns: Supplier · Invoice # · Amount · 4-leg match summary (icon + text) · Status pill (Variance / Verified payable / Processing). Rows clickable → Match Workspace.

### 2. Match Workspace · `/matching-engine/:invoiceId`
- Breadcrumb (AP Inbox / Supplier / Invoice)
- Header: supplier, invoice ref, amount, received date, line count; variance pill top-right when applicable
- 2×2 leg grid: PO · Receipt · Invoice · Vendor Agreement. Each card shows ref, key meta, match state (ok | warn)
- Line items table: line · SKU · qty · invoice unit · agreement unit · per-line pill. Variance lines highlight; clicking a variance line → Variance Evidence

### 3. Variance Evidence · `/matching-engine/:invoiceId/variance/:lineId`
- Breadcrumb
- Header: SKU + qty
- Four metric cells: Invoice unit · Agreement unit · Per-unit Δ · Total recoverable
- 2×2 evidence grid:
  1. SPA clause — ref + verbatim quote
  2. Signed agreement — counterparty + sign date + file ref
  3. PO line — PO ref + line meta
  4. Receipt line — receipt ref + qty meta
- Action row: `Route to Recovery` (primary) · `Override with reason` (secondary). Primary action creates a Recovery item and navigates to `/recovery`.

### 4. Recovery · `/recovery`
- Three stat cards: Open recoverable · Recovered MTD · Avg cycle
- Table: source invoice · supplier · variance type · amount · status pill (New · Credit memo pending · Recovered)
- Newly routed items show "New · routed today" pill for the session

## Data model
Supplier   { id, name }
Agreement  { id, supplierId, ref, signedDate, clauses: [{ ref, text, sku_class, max_unit_price, valid_through }] }
PO         { id, ref, supplierId, lines: [{ sku, qty, unit_price }] }
Receipt    { id, ref, poId, lines: [{ sku, qty_received, received_date }] }
Invoice    { id, ref, supplierId, received_date, total, lines: [{ sku, qty, unit_price }] }
Variance   { id, invoiceId, lineId, type, per_unit_delta, total_recoverable, status }

## Match logic
For each invoice line:
- PO match: SKU + qty present on linked PO
- Receipt match: SKU + qty present on linked Receipt
- Agreement match: invoice `unit_price` ≤ matching clause `max_unit_price` for the SKU class

If agreement match fails → emit Variance: `type = "SPA price breach"`, `per_unit_delta = invoice_unit − agreement_unit`, `total_recoverable = delta × qty`.
