Version 1.1

You are extending an existing Helm fintech platform (a React + TypeScript + Express + Drizzle ORM monorepo). The goal is to build out the full operational platform for Helm — an AI-native financial operations platform for mid-market wholesale distributors (HVAC, electrical) — covering the 12-stage distribution transaction lifecycle organised around 5 value creation tools (Recovery, Prevention, Visibility, Financing, Intelligence).

Critical constraint: extend the existing codebase, do not rebuild. 5. Backend — Extend server/routes.ts and server/storage.ts
For each new table, add:

MemStorage methods: get<Entity>, get<Entity>ById, create<Entity>, update<Entity>, list<Entities> (with filters where useful).
REST endpoints: GET /api/<entities>, GET /api/<entities>/:id, POST /api/<entities>, PATCH /api/<entities>/:id.
Domain endpoints (the interesting ones):
POST /api/expected-credits/compute — runs Helm's calc engine for a given program + period, returns expected credits per sale line.
POST /api/claims/:id/submit — moves a draft claim to submitted, generates EDI 844 payload (mock for now).
POST /api/claims/:id/adjudicate — accepts adjudication outcome (approved/partial/rejected with reason codes), creates settlement event.
POST /api/remittance/:id/apply — runs cash application against open AP, returns matched/unmatched lines.
POST /api/recon/four-way-match/:invoiceId — runs RECON #1 (4-Way Invoice Match) for an invoice, returns variance per leg.
POST /api/recon/settlement-audit/:claimId — runs RECON #2 for a settled claim, returns variance.
POST /api/recovery-advance/quote — given a set of expected credits, returns advance quote with fee.

8. Design System Constraints
Visual style
Dark mode is the default and primary aesthetic. Light mode optional.
Background: slate-950 / zinc-950 base. Cards on slate-900/50 with slate-800 borders.
Accent colour: emerald-500 for primary actions and positive metrics.
Status colours (existing pattern, keep consistent):
emerald: positive/recovered/approved
amber: pending/at-risk/warning
red: critical/denied/rejected
blue: informational/filed/submitted
slate: neutral/draft
Typography: Existing font stack. Use font-mono for amounts and IDs. Tabular-nums for tables.
Animation: Keep framer-motion use light. Animated stat counter on dashboards. Pulse on live items. No flashy entrances on page-level renders.
Charts: recharts only. Match the existing dark theme (axis lines slate-700, grid slate-800).
Maps: react-simple-maps with the existing countries-110m.json topojson.
Icons: lucide-react only.
Sidebar: Collapsible. Active route highlighted with emerald accent border-left.
Design tenets (cross-cutting product principles)
These are the non-visual rules that should be visible on most pages and tested in design review:

Show the gap. Every reconciliation surface displays an apparent figure, a reconciled figure, and the dollarised difference. The 4-Way Invoice Match shows agreement-vs-invoice. Settlement Audit shows the four-amount chain. Expected Credit Engine shows eligible-vs-claimed. Margin Attribution shows apparent-vs-true. Gap dollars must be visible without drilling in — that's the headline number on every page of this kind.

Line-level audit trail always reachable. Any number on any screen — a recovery total, an expected credit, a true margin — must be drillable in two clicks or fewer to the underlying ledger events that produced it. No black boxes. This is what makes the platform defensible to auditors and disputable suppliers.

Apparent figures should never stand alone. If the UI is showing gross margin, it should also be showing true margin. If it's showing claimed amount, it should also be showing expected amount. Standalone apparent numbers create the false confidence the platform exists to dismantle.

Master data issues surface where they cause harm, not where they live. A SKU mapping problem should appear on the Expected Credit Engine page (where it caused a line to drop) or the Unapplied Cash page (where it caused a remittance miss), not buried in the Master Data admin page. The admin page is for resolution, not detection.
